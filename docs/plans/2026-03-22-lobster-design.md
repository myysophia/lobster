# Lobster 设计文档

日期：2026-03-22

## 1. 项目目标

Lobster 是一个统一安装入口，用来降低 Claw 系产品的安装和首次使用门槛。

第一阶段只聚焦腾讯 WorkBuddy，目标不是替代官方安装器，而是在其外层增加一层更易用、更可诊断、更适合新手的命令行体验。

核心目标：

- 用户只需记住一个安装命令
- 用户不需要自己判断平台与安装文档路径
- 用户安装完成后能明确知道下一步做什么
- 常见失败场景能给出人话提示和修复建议

## 2. 产品定位

Lobster 不是 WorkBuddy 本体，也不是第三方 GUI 启动器。

它的定位是：

- 多产品统一安装入口
- 多平台安装编排器
- 首次体验引导器
- 基础诊断工具

第一阶段范围内，它服务于 WorkBuddy 的安装和启动体验优化。

## 3. 第一版策略

第一版采用“薄封装”策略。

含义：

- 复用腾讯官方安装脚本/安装器
- 不自建独立包分发与版本仓库
- 不接管 WorkBuddy 的内部配置格式
- 不承诺自动完成企业渠道绑定

这能最大程度降低维护成本，同时把精力集中在真正影响用户感知的部分：

- 更简单的命令入口
- 更稳定的安装结果检测
- 更友好的失败提示
- 更明确的下一步引导

## 4. 命令结构建议

为了兼容未来支持多种 Claw 产品，推荐采用“统一入口 + 产品参数”的命令体系。

推荐主形态：

```bash
lobster install workbuddy
lobster status workbuddy
lobster open workbuddy
lobster doctor workbuddy
lobster list
```

为了兼顾 WorkBuddy 第一阶段的易用性，可以提供快捷别名：

```bash
wb install
wb status
wb open
wb doctor
```

建议关系：

- `lobster` 是长期主入口
- `wb` 是 WorkBuddy 的快捷别名

如果后续进入更多产品，不需要推翻命令结构。

## 5. 语言与技术建议

推荐语言：`Go`

推荐原因：

- 适合实现跨平台 CLI 与系统命令编排
- 二进制分发简单
- 子进程调用、网络下载、文件检测实现成本低
- 后续跨平台发布链路成熟
- 团队维护成本相对更低

当前阶段不推荐 Rust，原因不是 Rust 不好，而是它更适合对系统层控制、安全和性能要求更高的长期产品，而不是第一版的轻量安装编排器。

## 6. 功能边界

第一版包含：

- 平台检测
- 调用官方安装器
- 安装结果校验
- WorkBuddy 启动入口
- 常见问题诊断
- 安装后下一步建议

第一版不包含：

- 自建 WorkBuddy 包管理体系
- 自动完成飞书/企微/钉钉绑定
- 修改 WorkBuddy 内部配置
- 离线安装包镜像
- GUI 安装器

## 7. 模块设计

推荐内部拆分为 5 个核心模块。

### 7.1 platform

职责：

- 检测当前操作系统
- 检测 Shell 与执行环境
- 识别当前是否具备桌面打开能力

### 7.2 installer

职责：

- 根据平台调用官方安装命令
- 记录安装执行结果
- 返回统一的安装状态

### 7.3 detector

职责：

- 检查 WorkBuddy 是否安装成功
- 检查可执行命令是否存在
- 检查应用路径是否存在
- 尝试获取版本或启动信息

### 7.4 launcher

职责：

- 在各个平台拉起 WorkBuddy
- 对无桌面环境场景做降级提示

### 7.5 advisor

职责：

- 把底层技术状态翻译成用户可理解的提示
- 给出下一步动作建议

## 8. 核心流程

`install workbuddy` 的推荐流程：

1. 预检查平台与网络
2. 检查是否已安装
3. 按平台选择官方安装路径
4. 执行安装
5. 校验安装结果
6. 输出明确下一步

“下一步”的设计原则：

- 如果已安装成功：告诉用户如何打开 WorkBuddy
- 如果未检测到 PATH 生效：提示重开终端
- 如果当前环境不支持桌面打开：提示后续手动启动方式

## 9. 错误处理原则

第一版的竞争力很大程度上来自错误处理体验，而不是安装本身。

优先覆盖的失败场景：

- 官方安装源不可达
- 权限不足
- PowerShell 执行策略拦截
- macOS 安全限制
- PATH 未刷新
- 桌面环境不可用
- 已安装但当前 shell 无法识别

错误提示要求：

- 明确说明现象
- 尽量说明原因
- 必须给出下一步操作建议

## 10. 扩展架构

Lobster 从一开始就应按“多产品安装器”设计。

建议引入产品注册表：

- `workbuddy`
- `arkclaw`
- `kimi-claw`
- `autoclaw`

每个产品都拥有自己的安装策略描述，但共享：

- 平台检测
- CLI 输出
- 错误处理框架
- 启动接口
- 诊断接口

## 11. 成功标准

第一版成功标准：

- 用户通过 `lobster install workbuddy` 或 `wb install` 成功安装 WorkBuddy
- 三端都能覆盖基础安装流程
- 用户不需要先读官网就知道下一步
- 常见错误能被工具解释
- 架构允许未来增加更多 Claw 产品

## 12. 当前建议结论

推荐结论如下：

- 项目名：`lobster`
- 第一版主目标：`WorkBuddy 安装体验重构`
- 技术路线：`Go + 薄封装`
- 命令策略：`长期主入口用 lobster，短期保留 wb 快捷命令`
